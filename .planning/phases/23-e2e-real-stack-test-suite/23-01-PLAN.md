# Plan 23-01 — Real Stack Infrastructure: LLM Stub + Docker Compose Test Stack + Fixtures

## Objective

Build the test infrastructure: docker-compose.test.yml overlay (separate test DB, LLM
stub service), FastAPI LLM stub server, Playwright Project Dependencies for setup/teardown,
real account fixture (worker-scoped), and internal DB reset endpoint.

See 23-RESEARCH.md for all verified patterns — use them exactly.

---

<task id="23-01-01" title="docker-compose.test.yml + LLM stub service">
  <objective>
    Create the Docker Compose test overlay and the FastAPI LLM stub server that the
    backend will hit instead of real Gemini/OpenAI during E2E tests.
  </objective>
  <read_first>
    - docker-compose.yml — existing services, env vars, network
    - docker-compose.override.yml — check for any existing dev overrides
    - api/app/core/config.py — Settings for LLM base URL
    - 23-RESEARCH.md — docker-compose.test.yml pattern + LLM stub code
  </read_first>
  <action>
    1. Create `docker-compose.test.yml` following the pattern in 23-RESEARCH.md:
       - Override `db` to use `senso_test` database
       - Override `api` env: `DATABASE_URL` → `senso_test`, `LLM_BASE_URL` → `http://llm-stub:4010`,
         `ALLOW_TEST_RESET=true`
       - Add `llm-stub` service with healthcheck
       - Use `condition: service_healthy` in all `depends_on` (not just `service_started`)
       - ALWAYS redeclare full `healthcheck:` block (never partial — Docker bug #10188)

    2. Create `api/tests/llm_stub_server.py` following 23-RESEARCH.md pattern:
       - Implements `/health`, `/v1beta/models/{model}:generateContent` (Gemini format),
         `/v1/chat/completions` (OpenAI format)
       - Fixture responses matched by substring in request body
       - Default fixture: valid `coaching_response` schema with `message`, all other fields null
       - Named fixtures: `"search_italy_rules"` trigger, `"get_user_profile"` trigger

    3. Create `api/tests/Dockerfile.llm-stub`:
       ```dockerfile
       FROM python:3.12-slim
       WORKDIR /app
       RUN pip install fastapi uvicorn httpx
       COPY llm_stub_server.py .
       CMD ["uvicorn", "llm_stub_server:app", "--host", "0.0.0.0", "--port", "4010"]
       ```

    4. Check `api/app/core/config.py` (or settings): ensure `LLM_BASE_URL` (or equivalent
       env var) is read by the LLM client. If the LLM client hardcodes the Gemini URL,
       add a `GEMINI_BASE_URL` override setting.
  </action>
  <acceptance_criteria>
    - `docker compose -f docker-compose.yml -f docker-compose.test.yml up --build llm-stub`
      starts the stub and `curl http://localhost:4010/health` returns 200
    - `docker-compose.test.yml` has full healthcheck blocks (not partial)
    - `llm_stub_server.py` parses Gemini request format and returns valid coaching JSON
  </acceptance_criteria>
</task>

<task id="23-01-02" title="Playwright project dependencies + real account fixture + DB reset">
  <objective>
    Add `real-stack` project config to playwright.config.ts, create global setup/teardown
    using Project Dependencies pattern, create worker-scoped real account fixture, add
    internal DB reset endpoint to FastAPI.
  </objective>
  <read_first>
    - senso/playwright.config.ts — existing projects config
    - senso/e2e/support/fixtures.ts — existing fixture pattern
    - api/app/routers/ — existing router registration
    - api/app/main.py — router mounts
    - 23-RESEARCH.md — exact patterns for all three components
  </read_first>
  <action>
    1. Add to `senso/playwright.config.ts` (following 23-RESEARCH.md Project Dependencies):
       ```ts
       { name: 'real-stack-setup', testMatch: /real-stack\/global\.setup\.ts/, teardown: 'real-stack-teardown' },
       { name: 'real-stack-teardown', testMatch: /real-stack\/global\.teardown\.ts/ },
       { name: 'real-stack', testDir: './e2e/real-stack',
         use: { baseURL: process.env.FRONTEND_URL || 'http://localhost:5173' },
         dependencies: ['real-stack-setup'] },
       { name: 'real-stack-mobile',
         testDir: './e2e/real-stack',
         use: { ...devices['iPhone 14'], baseURL: process.env.FRONTEND_URL || 'http://localhost:5173' },
         dependencies: ['real-stack-setup'] },
       ```

    2. Create `senso/e2e/real-stack/global.setup.ts` following 23-RESEARCH.md:
       poll API `/health` with `toPass({ intervals: [2000, 3000, 5000], timeout: 60_000 })`
       then POST to `/internal/db/reset` to clean state.

    3. Create `senso/e2e/real-stack/global.teardown.ts`: DELETE all `e2e-*@test.invalid`
       users via internal endpoint.

    4. Create `senso/e2e/real-stack/fixtures.ts` with worker-scoped `account` fixture
       from 23-RESEARCH.md: creates real user via `POST /api/auth/register`, yields
       `{ email, password, userId }`, tears down via `DELETE /internal/users/:id`.

    5. Create `api/app/routers/internal.py` (or add to existing internal router):
       - `POST /internal/db/reset` — guarded by `settings.allow_test_reset` + `X-Internal-Token` header
       - `DELETE /internal/users/{user_id}` — same guards
       - TRUNCATE tables: `users, user_profiles, uploads, transactions, coaching_sessions`
         with `RESTART IDENTITY CASCADE`
       - Mount in `main.py`: `app.include_router(internal_router, prefix="/internal")`

    6. Add `ALLOW_TEST_RESET=false` and `INTERNAL_TOKEN` to `api/app/core/config.py` Settings.
       Add to `.env.e2e.example`.
  </action>
  <acceptance_criteria>
    - `npx playwright test --project=real-stack-setup` runs setup without error
    - `POST /internal/db/reset` returns 403 without token, 200 with correct token
    - `DELETE /internal/users/{id}` removes the user from DB
    - Worker-scoped account fixture creates a real user account (verify in DB)
    - Playwright `real-stack` project listed in `--list` output
  </acceptance_criteria>
</task>
