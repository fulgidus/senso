## Objective

Three backend changes that collectively ensure no admin user can have `username = "!admin"` or `username IS NULL`, and that the `POST /admin/claim-handle` endpoint enforces strict format rules on the handle value:

1. **D-09** — Fix `signup()` so new admin accounts receive a `$adj-noun-N` username instead of `"!admin"`.
2. **D-10** — Add Round 22 to `_add_missing_columns()` that backfills existing admin users who have `username IS NULL` or `username = '!admin'` with a unique `$adj-noun-N`.
3. **D-11** — No new endpoint needed; the claim-handle endpoint already exists.
4. **D-12** — Tighten `ClaimHandleRequest` with a `@field_validator` enforcing: must start with `!`, stripped value matches `^[a-z0-9-]+$`, stripped length 3–30, stripped value not in reserved list `{"admin", "sistema", "senso"}`.

A Wave 0 test file is created first to lock regression expectations.

## must_haves

- `generate_admin_username()` is no longer called in `auth_service.py`
- New admin signup produces a `$adj-noun-N` username (verified by test)
- Existing admins with `username IS NULL` or `username = "!admin"` get backfilled (verified by test)
- `POST /admin/claim-handle` with `"!admin"` returns 422 (verified by test)
- `POST /admin/claim-handle` with uppercase or spaces returns 422 (verified by test)
- `POST /admin/claim-handle` with duplicate handle returns 409 (verified by test)
