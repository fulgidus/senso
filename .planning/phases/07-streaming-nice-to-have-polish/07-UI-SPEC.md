## Spacing Scale

| Token | Value | Usage |
| --- | --- | --- |
| xs | 4px | Streaming cursor gap, inline icon spacing, chip interior spacing |
| sm | 8px | Compact button gaps, toast content padding, persona row inner spacing |
| md | 16px | Default bubble padding, modal row spacing, settings control spacing |
| lg | 24px | Header/subheader padding, card padding, modal body padding |
| xl | 32px | Vertical gaps between chat sections and settings groups |
| 2xl | 48px | Major modal/sheet section breaks |
| 3xl | 64px | Reserved for empty-state screens only |

Exceptions: interactive tap targets must be at least **44px** high/wide; persona list rows must be at least **48px** high; restore toast minimum height **40px**.

## Color

Use existing CSS variables from `senso/src/index.css`; do not hardcode raw colors in component logic.

| Role | Value | Usage |
| --- | --- | --- |
| Dominant (60%) | `--background` (`#FFFFFF` light / `#112D4E` dark) | App background, chat canvas, settings page background |
| Secondary (30%) | `--card` / `--secondary` (`#DBE2EF` light / `#1A3A5C`-`#1E4472` dark) | Assistant bubbles, settings cards, picker surfaces, history modal panels |
| Accent (10%) | `--primary` (`#3F72AF` light / `#5B9BD5` dark) | Selected persona trigger, send action, active settings choice, focus ring, restore toast accent |
| Destructive | `--destructive` token | Delete conversation, log out, destructive confirmation only |

Accent reserved for: send button, selected persona state, selected default-persona state in Settings, keyboard focus rings, restore toast icon/border, retry action link. Never use accent for all assistant bubbles or all tertiary controls.

**Visual hierarchy contract:**
- Primary focal point: the live assistant streaming bubble in the main chat column.
- Secondary focal point: the composer and send/voice controls directly below the conversation.
- Subordinate elements: restore toast, persona switcher, and history affordances must stay visually quieter than the active assistant reply.
- Structured cards and `details_a2ui` may draw attention only after the streamed message completes; they never outrank the live reply while it is in progress.

## Interaction Contract

### 1. Streaming assistant reply

- Append the user bubble immediately on submit.
- Mount an assistant bubble shell within **300ms** of submit.
- The shell must be a real assistant bubble, not the current multi-line skeleton block.
- Until the first chunk arrives, show only a subtle typing treatment: three muted dots + `coaching.thinking.*` label.
- Stream only `response.message` text. Do **not** stream `action_cards`, `resource_cards`, `learn_cards`, `affordability_verdict`, or `details_a2ui`.
- When text is streaming, keep the bubble height stable with normal text wrapping; do not reserve fake card placeholders.
- When the final structured payload lands, insert structured sections below the completed text with **180ms** fade/slide-up, ease-out.
- If SSE is unavailable or breaks mid-reply, finish the same reply via the existing full-response path in the same bubble. No modal, no banner, no extra retry prompt.
- Voice mode may show streaming text first, but TTS autoplay starts only after the final reply text is complete.
- The play-audio button stays disabled/hidden until the final text is available.

### 2. Scroll and history behavior

- On returning to chat, auto-open the newest session and load the **full visible history** before rendering the composer as ready.
- After restore completes, scroll to the latest message once.
- During an active stream, auto-scroll only if the user is already near the bottom (within ~120px). If the user scrolls upward, do not yank the viewport.
- Keep the current history button/modal entry point for older conversations. No permanent sidebar in this phase.

### 3. Restore toast

- Show a non-blocking toast immediately after a session is auto-restored.
- Position: below the chat subheader, centered on mobile and desktop.
- Style: compact pill/card, secondary background, 1px accent border, label typography.
- Motion: **180ms** fade/slide in, visible **2500ms**, **150ms** fade out.
- Accessibility: `role="status"`, `aria-live="polite"`.

### 4. Quick persona switcher in chat

- Use a compact outline trigger in the chat header with persona icon + current persona name + chevron.
- On tap/click, open a bottom-sheet style picker on mobile and the same centered modal pattern on desktop for consistency.
- Each persona row contains: icon, name, one-line description, selected checkmark.
- Each row uses the persona tint as a faint background only when selected.
- Switching persona updates only **future** assistant replies. Existing messages keep their original persona styling and stored persona metadata.
- Starting a **new conversation** must use the saved default persona from Settings, not the last temporary switch.

### 5. Persona cues in message history

- Keep persona visibility subtle: no loud character cards.
- Assistant bubbles use persona identity through three cues only: avatar background tint, 1px bubble border tint, optional tiny label row (`icon + name`) at the top of the bubble on the **first assistant message after a switch**.
- Subsequent assistant messages in the same uninterrupted persona streak rely on tint alone.
- Restored history must preserve the correct tint per assistant message based on stored `persona_id`.

### 6. Persistent default persona in Settings

- Add a new **Default coach** settings group above the existing voice gender/auto-listen controls.
- Render personas as stacked selectable rows inside the existing settings card pattern.
- Selected row treatment: accent outline + persona tint wash + trailing check icon.
- Save behavior follows the existing Settings save model: changes are local until the user presses Save.
- Saving the default persona should reuse the existing success/error inline feedback pattern already used in Settings.

### 7. Destructive actions

- Deleting a conversation from history requires confirmation before deletion.
- Use a compact confirmation dialog/sheet for delete with two actions: `Keep conversation` and destructive `Delete conversation`.
- Logging out keeps the current confirmation behavior and destructive styling, but uses exact labels `Stay signed in` and destructive `Log out`.
- Generic secondary labels such as `Cancel` are not allowed for destructive flows in this phase.

### 8. Motion and accessibility

- Respect `prefers-reduced-motion`: disable simulated typing animation, reduce toast/bubble motion to simple fades, and insert final structured UI without slide movement.
- Streaming bubble container should use `aria-live="polite"`; do not announce every token aggressively.
- Persona picker rows must be fully keyboard reachable and visibly focused.
- Color alone cannot indicate selected persona; always pair tint with a checkmark or active border.
