# Plan 21-02 — Remove Fallback Injection + Surface Caps

## Objective

Remove `_inject_fallback_cards()` unconditional behavior and apply hard caps:
resource_cards ≤ 2, action_cards ≤ 3. Gate resource_cards on `search_content` actually
having been called during the response. This eliminates the "every response looks like
a wall of cards regardless of intent" problem.

---

<task id="21-02-01" title="Gate resource_cards on tool call + remove unconditional injection">
  <objective>
    `_inject_fallback_cards()` currently fires whenever `resource_cards` is empty,
    injecting irrelevant content. Remove or gate this so resource_cards only populate
    when `search_content` was actually called and returned scored results.
  </objective>
  <read_first>
    - api/app/coaching/service.py — `_inject_fallback_cards()`, `_tool_executor()`,
      tool call tracking, response assembly
  </read_first>
  <action>
    1. In `_tool_executor()`, track whether `search_content` was called:
       ```python
       self._search_content_called = False  # reset per request

       # Inside search_content handler:
       self._search_content_called = True
       ```

    2. Modify `_inject_fallback_cards()` or its call site:
       - If `purchase_intent = True` and `search_content` was NOT called:
         call fallback injection (limited to 2 cards)
       - If `purchase_intent = False`: never inject fallback cards
       - If `search_content` WAS called: resource_cards already populated from tool result

    3. Gate: `resource_cards` in the final response is populated ONLY from:
       - LLM emitted them after calling `search_content`
       - Fallback injection (only on purchase intent when search wasn't called)
       Never from schema repair / default fill.
  </action>
  <acceptance_criteria>
    - Informational question ("Cos'è ISEE?") → `resource_cards: []`
    - Purchase question where LLM didn't call `search_content` → max 2 fallback cards
    - Purchase question where LLM called `search_content` → cards from tool result, capped at 2
    - Pytest: `test_surface_caps.py::test_no_fallback_on_informational`
  </acceptance_criteria>
</task>

---

<task id="21-02-02" title="Hard caps: resource_cards ≤ 2, action_cards ≤ 3">
  <objective>
    Enforce hard caps on both card arrays before the response is returned, regardless
    of how many the LLM emitted.
  </objective>
  <read_first>
    - api/app/coaching/service.py — response DTO assembly
    - api/app/coaching/schemas/coaching_response.schema.json — maxItems constraints
  </read_first>
  <action>
    1. In response assembly (before serializing to DTO):
       ```python
       response["resource_cards"] = response.get("resource_cards", [])[:2]
       response["action_cards"] = response.get("action_cards", [])[:3]
       ```

    2. Add `"maxItems": 2` to `resource_cards` array in `coaching_response.schema.json`
       and `"maxItems": 3` to `action_cards`. This enforces it at schema level too.

    3. Also cap `new_insight` to 1 (already usually singular — add explicit `maxItems: 1`
       or truncate list to first item if LLM returns array).
  </action>
  <acceptance_criteria>
    - Even if LLM returns 5 resource_cards, API response has at most 2
    - Even if LLM returns 6 action_cards, API response has at most 3
    - Schema validates with `maxItems` constraints
    - Pytest: `test_surface_caps.py::test_resource_cap`, `test_action_cap`
  </acceptance_criteria>
</task>
