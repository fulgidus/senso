# Plan 21-01 — Purchase Intent Classifier + Conditional Schema Selection

## Objective

Add a lightweight (no LLM) purchase intent classifier to `CoachingService`. When the
user's message does not contain purchase intent, send a stripped schema to the LLM that
omits `affordability_verdict` — preventing the current false-positive verdicts on
informational questions ("Cos'è l'IRPEF?" → verdict: "no").

---

<task id="21-01-01" title="Purchase intent classifier">
  <objective>
    Add `_detect_purchase_intent(message: str) -> bool` to `service.py`. Uses regex +
    keyword matching only — zero LLM cost, runs before the LLM call.
  </objective>
  <read_first>
    - api/app/coaching/service.py — existing `_sanitize_input()` and chat flow
  </read_first>
  <action>
    Add to `CoachingService`:
    ```python
    import re

    _PURCHASE_PATTERNS = re.compile(
        r'\b(posso comprare|mi conviene|ho i soldi per|budget per|afford|conviene acquistare'
        r'|costo di|posso permettermi|posso pagare|posso prendere|mi posso permettere'
        r'|ho abbastanza per|vale la pena comprare|comprare|acquistare|pagare)\b',
        re.IGNORECASE
    )

    def _detect_purchase_intent(self, message: str) -> bool:
        """Return True if message contains purchase/affordability intent."""
        return bool(self._PURCHASE_PATTERNS.search(message))
    ```

    Also add a simple fallback: if the message ends with a question mark AND is ≤ 6 words,
    treat it as informational (not purchase intent) regardless of keyword matches.
  </action>
  <acceptance_criteria>
    - "Posso comprare un iPhone?" → True
    - "Mi conviene il fondo pensione?" → True
    - "Cos'è l'IRPEF?" → False
    - "Spiega il TFR" → False
    - "Ho abbastanza risparmi?" → True
    - "Cosa sono i contributi INPS?" → False
    - Pytest: `test_intent_classifier.py` with ≥ 15 labeled examples, accuracy ≥ 90%
  </acceptance_criteria>
</task>

<task id="21-01-02" title="Conditional schema selection (simple vs full response)">
  <objective>
    When purchase intent is False, send `coaching_simple_response.schema.json` (which
    lacks `affordability_verdict`) instead of the full schema. Ensures the LLM cannot
    emit a verdict on informational questions.
  </objective>
  <read_first>
    - api/app/coaching/service.py — `_build_coaching_request()`, schema loading
    - api/app/coaching/schemas/coaching_response.schema.json — full schema
    - api/app/coaching/schemas/coaching_simple_response.schema.json — stripped schema
  </read_first>
  <action>
    1. Update `coaching_simple_response.schema.json`: ensure it is a copy of
       `coaching_response.schema.json` with `affordability_verdict` field removed from
       `properties` and `required` array, and `additionalProperties: false`.

    2. In `CoachingService.chat()`:
       ```python
       intent = self._detect_purchase_intent(sanitized_message)
       schema_name = "coaching_response" if intent else "coaching_simple_response"
       schema = loader.get_schema(schema_name)
       ```

    3. In the response DTO assembly: if `affordability_verdict` is missing from LLM
       output (simple schema used), set it to `null` explicitly (never fill with default).

    4. Track `purchase_intent` in the response context for surface gating in Plan 21-02.
  </action>
  <acceptance_criteria>
    - Informational message → LLM receives `coaching_simple_response.schema.json`
    - Informational LLM response has `affordability_verdict: null` in API response
    - Purchase message → LLM receives full `coaching_response.schema.json`
    - Pytest: `test_intent_classifier.py::test_schema_selection_informational` and
      `test_schema_selection_purchase`
  </acceptance_criteria>
</task>
