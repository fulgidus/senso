## Objective

Fix `_repair_response()` to coerce `null` array fields to `[]`, add `content_cards` and `interactive_cards` to the schema `required` array, and add a regression test for the null coercion.

## Background

`dict.setdefault(key, default)` only inserts the default when the key is **absent**. When the LLM returns `"content_cards": null`, the key exists with value `None`, so `setdefault` is a no-op and the null passes through to the API response. The frontend then crashes on `resp.content_cards.length` (null dereference on line 710 of ChatScreen.tsx).

The schema already uses `"type": "array"` (not `["array", "null"]`) for both fields, so D-04 (type change) is a **no-op** — only the `required` array entry is missing. This finding is documented in task 26-01-T2.

```python
def test_repair_response_coerces_null_arrays(self):
    """_repair_response must coerce null content_cards/interactive_cards to []."""
    from app.coaching.service import CoachingService
    svc = CoachingService.__new__(CoachingService)
    repaired = svc._repair_response({
        "message": "test",
        "content_cards": None,
        "interactive_cards": None,
    })
    assert repaired["content_cards"] == []
    assert repaired["interactive_cards"] == []
```

Note: `CoachingService.__new__(CoachingService)` creates an instance without calling `__init__`, which is the same pattern already used by `test_repair_response_fills_missing_fields`. Mirror that exact instantiation pattern.
