## Objective

Add null-safe `??` and `??=` guards in `ChatScreen.tsx` so the frontend never crashes when `content_cards` or `interactive_cards` arrive as `null` from the API or from stored history messages.

## Background

Even after Plan 26-01 fixes the backend, the frontend must be hardened as a belt-and-suspenders defence for:
1. **Live API responses:** `resp.content_cards.length` (line 710) and `resp.interactive_cards.length` (line 715) crash if either field is `null` — this can happen from race conditions, API errors, or partial schema migration rollout.
2. **History messages:** `parseStoredMessage()` parses stored `SessionMessage.content` via `JSON.parse` and casts it to `CoachingResponse`. Messages stored before the backend fix may have `content_cards: null`; without normalization, replaying history would crash on `.length`.
