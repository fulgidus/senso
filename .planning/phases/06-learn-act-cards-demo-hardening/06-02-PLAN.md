# Plan: MARP Visual QA + Speech-to-Speech End-to-End Fix

## Goal

Confirm that MARP slide decks render correctly in the browser and that the full voice loop (STT → coaching API → TTS → audio playback) works as a connected end-to-end flow, fixing any bugs found.

## Context

Both features are built but untested as integrated wholes:

**MARP slide viewer:**
- `MarpSlideViewer.tsx` parses `.md` files from `SLIDE_INDEX`, renders HTML via `marked`, shows prev/next navigation and fullscreen toggle, applies `senso-light`/`senso-dark` CSS classes
- CSS is defined in `index.css` (`.marp-viewer`, `.marp-slide`, `.marp-nav`, theme classes)
- 6 slide decks exist: `it-slide-budget-base`, `it-slide-tan-taeg`, `it-slide-fondo-emergenza`, `it-slide-etf-intro`, `en-slide-budget-basics`, `en-slide-compound-interest`
- `ResourceCardRouter` routes to `MarpSlideViewer` only when `card.resource_type === "slide_deck" && card.slide_id` is truthy
- The `slides.json` catalog has no `slide_id` field — the `id` field IS the slide ID. The LLM must be instructed to set `slide_id = <catalog id>` in the resource card

**Speech-to-speech loop:**
- `useVoiceInput` (Web Speech API) → `onFinalTranscript` → `handleSend()` in `ChatScreen`
- `handleSend()` → `sendMessage()` → coaching API → response appended to messages
- `onAssistantMessageRef.current(response.message)` called in `handleSend` → triggers `useTTS.play()`
- `useTTS` calls `fetchTTSAudio()` (POST `/coaching/tts`) → falls back to `window.speechSynthesis` on 503/error
- Known risk: `onAssistantMessage` is wired via a ref to break circular hook dependencies; timing issues during loading state teardown could cause TTS to fire before `isLoading=false`
