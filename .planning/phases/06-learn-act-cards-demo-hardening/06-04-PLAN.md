# Plan: Loading States + Error Recovery Polish

## Goal

Add visible skeleton/spinner loading states while the coaching API is generating a response, and harden graceful error recovery for LLM timeouts and TTS failures so the demo never looks frozen or broken.

## Context

**What exists:**
- `isLoading` state is already tracked in `ChatScreen.tsx`
- A basic "thinking" pulse text is shown during `isLoading`: `<span className="animate-pulse">{t('coaching.thinking.masculine')}</span>` — this is already wired (lines 1085-1092)
- Error display: `error` state renders a destructive banner at the bottom when `CoachingApiError` is caught
- `getErrorMessage()` maps error codes to i18n strings including `llm_error` and `network_error`
- TTS: `useTTS` already falls back to `speechSynthesis` on ElevenLabs failure; no user-facing error is shown
- Backend: `CoachingService.chat()` raises `CoachingError("llm_error", ...)` with status 502 when the LLM fails; the API endpoint converts this to HTTP 502

**What needs improving:**
- The "thinking" bubble is a single pulsing sentence — not enough visual weight for a 5-15 second wait
- No timeout on the frontend `sendMessage()` call (the backend has 60s LLM timeout; if the API itself hangs, the frontend waits indefinitely)
- The error banner disappears when the next message is sent but stays visible indefinitely otherwise — should auto-dismiss after a few seconds for non-fatal errors
- No explicit user feedback when TTS generation is slow (there is a spinner on the play button but not on auto-play in voice mode)
- No retry affordance on the error banner for transient LLM failures

   ```tsx
   const timeoutId = setTimeout(() => {

     clearTimeout(timeoutId)
     // ... existing error handler
   } finally {
     setIsLoading(false)
   }
   ```
